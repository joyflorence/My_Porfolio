Sales Insights Dashboard

Repository dashboard

Context

Mid-size omnichannel retailers generate plenty of transactional data but have no automated way to turn it into a timely decision. Revenue gets prioritized over profitability by default, simply because margin isn't visible in the moment a decision is made. This often make it hard to track performance and put more areas that need marketing and products in high demand based on seasons.

Role & constraints

Independently built, end to end. Working with transactional structured data got from kaggle - this dataset only has sale price, not cost so there is no real profit number to compute. The etl pipeline generates a simulated unit cost (a random sale price) specifically so the margin/profitability analysis in this projecto is demonstrated. This project was built over a period of two weeks

Approach

Built the pipeline in layers rather than the dashboard first: raw ingestion → PostgreSQL warehouse modeling → KPI definitions → Streamlit front end → Slack alerting on margin drops. Each layer was testable on its own before the next one depended on it.

A decision worth explaining

The harder problem wasn't the ETL, it was deciding what "margin drop" should mean - a raw percentage point threshold produced too many false alerts on naturally noisy days, so the alert logic compares against a rolling baseline instead of a fixed cutoff.

Result

A working pipeline that ingests sales data, loads it into a PostgreSQL warehouse, and surfaces KPIs with automated margin-drop alerts — you can try the live dashboard above with the sample data loaded.

Sentiment Analysis

Repository

Context

The goal was to turn an unstructured public webpage (a Wikipedia table) into structured, analyzable data, then apply basic text mining to surface patterns in one of the fields a compact demonstration of the full "scrape → clean → structure → analyze" pipeline.

Approach

Data acquisition — Used requests + BeautifulSoup to pull and parse the HTML of the Wikipedia page "List of largest companies in the United States by revenue". Table extraction — Located the target table element, then pulled out header cells to build column names (Rank, Name, Industry, Revenue (USD millions), Revenue growth, Employees, Headquarters). Row parsing & cleaning — Iterated over all tr rows, extracted cell text, stripped whitespace, and appended each row into a pandas DataFrame — effectively hand-rolling an HTML-table-to-DataFrame converter. Persistence — Exported the cleaned DataFrame to Largest companies in USA.csv, then reloaded it to confirm integrity (a good "checkpoint" habit for reproducibility). Text mining — Concatenated the entire Headquarters column into one text blob and fed it to the WordCloud library, visualized with matplotlib, to surface which cities/states appear most frequently among top revenue-generating companies.

Result

Successfully scraped and structured 100 rows × 7 columns of company data (Walmart, Amazon, Apple, UnitedHealth Group, Berkshire Hathaway,down to Best Buy at #100), including revenue, revenue growth %, employee counts, and HQ location. Produced a clean, reusable CSV dataset from a live web source. Generated a word cloud of company headquarters locations, visually highlighting the geographic concentration of America's largest companies (e.g., clusters like California, New York, Texas locations appearing more prominently based on frequency).
